# Governance

This document describes how the Bridgelet project is governed.
It aims to ensure transparency, clarity, and inclusive participation as the project evolves.

The governance model described here reflects the current state of the project and may
change as the community grows.

---

## Governance Structure

Bridgelet follows a **maintainer-led, community-informed** governance model.

Decisions are made by maintainers with input from contributors and the broader community.

---

## Roles and Responsibilities

### Contributor
Anyone who participates in the project by:
- Opening issues
- Submitting pull requests
- Improving documentation
- Providing feedback or ideas

Contributors do not have merge rights.

---

### Committer
Trusted contributors who:
- Consistently provide high-quality contributions
- Understand the project architecture and goals

Committers may be granted limited write access to specific areas of the repository.

---

### Maintainer
Maintainers are responsible for:
- Project direction and roadmap
- Reviewing and merging pull requests
- Managing releases
- Resolving conflicts
- Upholding project values and code of conduct

Maintainers have full repository access.

---

## Becoming a Maintainer

A contributor may be invited to become a maintainer based on:
- Sustained, high-quality contributions
- Demonstrated understanding of Bridgelet and Stellar
- Constructive participation in discussions
- Alignment with project goals and values

Promotion is decided by existing maintainers through consensus.

---

## Decision-Making Process

### Proposals
- Proposals are made via GitHub Issues or Pull Requests
- Larger changes should include a clear problem statement and rationale

---

### Approval
- Minor changes may be approved by a single maintainer
- Significant changes require discussion and consensus among maintainers

Formal voting is not currently used; decisions are consensus-based.

---

### Roadmap Prioritization
Roadmap priorities are determined by maintainers based on:
- User needs
- Technical feasibility
- Project goals
- Community feedback

---

## Project Operations

### Release Management
- Releases are managed by maintainers
- Versioning follows semantic versioning where applicable
- Release notes are published for each release

---

### Conflict Resolution
In the event of disagreements:
1. Discuss respectfully in the relevant issue or PR
2. Seek clarification and compromise
3. Escalate to maintainers if unresolved

Maintainers have final decision authority.

---

### Communication Channels
Project communication occurs primarily through:
- GitHub Issues
- GitHub Pull Requests
- Project documentation

Additional channels may be added as the community grows.

---

### Meetings
There are currently no scheduled governance meetings.
Decisions are made asynchronously.

---

## Transparency and Inclusivity

Bridgelet aims to:
- Document decisions when possible
- Encourage community input
- Keep governance processes open and understandable

Governance practices will evolve as the project matures.
